# Gromozeka — Roadmap и отложенные идеи

## Roadmap

### Phase 1 — Stride Mode (текущий фокус)

1. **Планирование** — LLM разбивает сообщение на шаги через `create_plan`
2. **Выполнение** — app управляет циклом через tool_result, LLM работает над шагами
3. **Управление планом** — `update_plan`, `request_user_input`, прерывание

Результат: работающий pipeline "написал сообщение → план → выполнил → итоги".

### Phase 2 — Кристаллизация (DRAFT)

4. **Шаги → триплеты** — после выполнения шаг превращается в долгосрочные факты (subject → predicate → object)
5. **Триплеты → Knowledge Graph** — entity обогащаются новыми связями, шаги удаляются

Результат: система накапливает знания между сессиями.

### Phase 3 — Reasoning, Leolani-style (DRAFT)

6. **Graph patterns** — при добавлении триплетов проверка на:
   - **Novelty** — новый факт → сохранить
   - **Conflict** — противоречие → создать `correct` или `query` шаг
   - **Gap** — неполный entity → создать `query` шаг
   - **Complement** — дополнение → обогатить
7. **Thoughts → новые шаги** — паттерны графа порождают новые DU, которые возвращаются в Phase 1

Результат: система сама замечает пробелы и противоречия в знаниях, задаёт вопросы, уточняет.

## Отложенные идеи

Собраны по ходу проектирования. Не для текущей реализации — вернуться когда Phase 1 будет стабильно работать.

### Дистилляция

После step_complete app может заменять развёрнутый result компактным дистиллятом. Не суммаризация (структура процесса), а извлечение сути. Сохраняет: результат, решения, изменения в entities. Выбрасывает: промежуточные tool calls, поисковые запросы, черновики. Точка вставки: между step_complete и подачей следующего шага в tool_result.

### Каскадная инвалидация

Если шаг инвалидируется — зависимые по `depends_on` тоже протухают. Семантика depends_on шире чем "жди результат": включает "основан на факте" (command → inform). Механизм не проработан — нужно решить: app автоматически или через update_plan.

### ReAct / Reflexion / LATS — execution внутри шага

Сейчас LLM свободно вызывает tools внутри шага. В будущем — формализовать как ReAct loop (Thought → Action → Observation) с явной рефлексией при неудаче (Reflexion) и tree search для сложных решений (LATS). Это будет собственная state machine внутри шага.

Ссылки:
- **ReAct** (Yao et al., 2023) — [arXiv:2210.03629](https://arxiv.org/abs/2210.03629)
- **Reflexion** (Shinn et al., 2023) — [arXiv:2303.11366](https://arxiv.org/abs/2303.11366)
- **LATS** (Zhou et al., 2023) — [arXiv:2310.04406](https://arxiv.org/abs/2310.04406)
- **BDI** (Rao & Georgeff, 1995) — маппинг типов шагов на Belief-Desire-Intention

### Обязательный Context Gathering

Сейчас: часть инструкции к шагу ("собери контекст"). В будущем: принудительная фаза перед работой над шагом — graph_query, поиск файлов, web search. Не опциональный "может поискать", а обязательный первый этап.

### Approval плана

Перед началом execution показать план пользователю для подтверждения. Варианты: автоматический запуск (как сейчас) vs approval через request_user_input.

### Passthrough оптимизация

Для inform/correct/evaluate/commit/condition — app может сам пометить COMPLETED без LLM call (просто graph_store). Экономия на API вызовах для простых шагов.

### Сменные промпт-инструкции

Архитектура "агент + сменная инструкция": system prompt = роль агента, user message = тред + инструкция задающая формат вывода. Разные инструкции для планирования (semantic decomposition), кристаллизации (триплеты), и других режимов.

### Кристаллизация (Phase 2)

Преобразование результатов шагов в триплеты: (subject, predicate, object) + perspective (polarity, certainty, sentiment). Правила: только подтверждённые факты, предикаты как глаголы с предлогами, conflict detection.
